SafeCache设计

发表于 2019-03-08 | 分类于 类

现状 & 需求

当前问题：

  • 在大部分记录数据场景中，数据记录后会先缓冲到内存中，之后再转移到持久化存储中，如果在转移前发生崩溃或应用退出，则内存中数据会丢失，为了防止丢失需实时写入，而这样会造成频繁的IO操作影响性能
  • 多进程情况一般会有如配置文件共享之类的需求

mmap具有以下特性：

  • 较高的IO性能（与内存直接读写等级相当，远优于普通io）
  • 存于其中的内容在进程崩溃后会由系统负责写回磁盘

目标

基于当前现状，拟以mmap为基础实现通用的安全缓存SafeCache，SafeCache具有以下特点：

  • 以较高的性能保证数据安全
  • 支持多进程数据共享

SafeCache使用场景为：

  • 可替换目前使用的内存缓存 + 持久化策略中的内存缓存，提供安全的数据功能
  • 作为快速、轻量的持久存储使用
  • 多进程数据共享使用

自定义数据类型

对于基础的数据类型，SafeArrayList和SafeHashMap内部会自动进行序列化和反序列化操作，
如要使用非基本类型对象，在初始化SafeArrayList、SafeHashMap时需传入实现IConverter接口的自定义 DataConverter

实现

容器类

SafeArrayList 内部维护一个 ArrayList(内存中) 和一个map文件映射

  • 初始化时使用Mmap中内容填充List
  • 增删改查时内存中直接修改ArrayList，另外同步修改Mmap
  • Mmap 采用线性操作，基础操作只支持追加写入操作，通过策略实现增、删、改

SafeHashMap 内部维护一个HashMap(内存)和一个map文件映射

  • 初始化时使用Mmap中内容填充Map
  • 增删改同SafeArrayList

数据结构

安全缓存数据以线性方式存放于磁盘中，内部使用流式存储方式实现，支持的基本操作为

  • 持续追加
  • 整体重写

增删改查功能通过相关策略实现，总体数据结构如下：

  • 信息文件
IHD LK WC EP CC FL FV
4byte 1byte 4byte 4byte 4byte 4byte 4byte

字段说明如下：

IHD：信息文件文件头
LK：文件锁区域
WC: 写入次数,次数增加说明内容改变
EP: 数据文件有效数据结尾位置，改变后需要加载更新
CC: clear计数,变化说明发生清空,根据fv判端要不要重map文件
FL: 文件长度
FV: 文件版本,数据文件的后缀号

  • 主数据文件和临时文件(重整时会写到新的临时文件，写入完成后设置为主文件)
FHD M item sum … M item sum
4byte 2byte 1byte 2byte 1byte

各字段说明如下：

FHD：文件头, 4byte, {‘s’, ‘a,’ ,’f’, ‘i’}
M：数据开始标识，内容固定，用于数据区分
item：真正的数据部分
sum: item内容校验和

item部分结构如下：

KLEN KDATA VLEN VDATA
4byte 4byte

各字段说明如下：

KLEN：4字节，表示KEY部分长度，对应map的键，list的索引
KDATA：实际的数据，List中对应一个Data序列化的内容，Map中可组装k-v为一个data
VLEN：4字节，表示VALUE部分长度
VDATA：内容

增删改查实现

对安全缓存的数据操作，实质都是在映射文件上append一个数据块（item），其他进程同步数据时会读取新的数据块，根据块内容判断需要进行的操作。

不同操作对应的数据块如下：

  • 增：添加一个正常的 item
  • 删：添加一个 key 为要删除的 key, VLEN 为 -1, VDATA 为空的item
  • 改：同增
  • 查：查询内存副本中的数据，如无内存副本无法查询
  • 清除：清除所有数据，info中 cc+1、ep = FIRST_DATA_PTR

状态判断

多进程共享同一数据文件时，每个进程内会维护自身状态信息，每次数据操作前会判断info文件中的状态与自身状态关系，从而判断出有没有数据更新和数据发生了何种更新，状态判断更新判断方式和过程如下：

更新状态分为 UPDATE 和 RESET，两种状态对应的处理为：

  • UPDATE: 从自身记录的数据尾指针开始读取到真正尾指针位置，使用读取到的item更新自身数据
  • RESET: 清除自身数据，重新重头加载数据

假设有两个进程A、B，两进程初始状态如下：

ProcessA aptr
Data item1 item2 item3 item… itemN
ProcessB bptr

Aptr表示A记录的自身数据尾指针位置，Bptr为B进程数据自身数据尾指针位置

数据更新 update

当A进程写入新数据后状态如下：

ProcessA aptr
Data item1 item2 item3 item… itemN itemN+1 itemN+2
ProcessB bptr

B进程再次操作数据时会判断出状态为 update，这时B会读取itemN+1、itemN+2来对内存中的数据结构进行增删改操作，操作完成后A和B状态如下：

ProcessA aptr
Data item1 item2 item3 item… itemN itemN+1 itemN+2
ProcessB bptr
数据重整 reset

当A进程写数据后空间不够，需要重新完整写入，或者A进程了clear操作，之后写入，数据状态如下：

ProcessA aptr
Data item1 item2
ProcessB bptr

B进程再次操作数据前，会判断出状态为 reset，这时B会清除内存中的数据，然后将Bptr置于初始位置，之后读取出item1、item2，并更新内存中的数据，B进程更新后数据状态如下：

ProcessA aptr
Data item1 item2
ProcessB bptr
读写同步

进程内的同步使用 synchronize 方式，进程间使用文件锁方式控制：

  • 进程需要读时，需先获取共享锁，然后读，读完成释放锁
  • 进程需要写时，需先获取共享锁，再获取独占锁，然后写，写完成释放锁

获取相关锁时可能存在等待，相关操作需放在线程中处理。文件锁在进程崩溃后可自动释放
